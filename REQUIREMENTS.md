# WinLauncher 要件定義

## 概要

Windows 11 向けキーボードランチャー（Tauri v2 + React + Tailwind CSS + TypeScript）

## 機能要件

### 基本動作

- グローバルホットキー（デフォルト Alt+Space、設定画面から変更可能）で検索ウィンドウをトグル表示
- インクリメンタル検索で設定済みフォルダ内のファイルを走査し絞り込み（初期値: なし）
- Enter で選択アプリを起動
- 検索ボックスに数式（例: 1000 + 1000）を入力すると、ファイル検索結果とは別枠の固定表示領域に計算結果をカンマ区切りでリアルタイム表示する。数式らしい入力であってもファイル検索（`search_files`）は従来通り実行され、ファイル検索結果は通常通り選択・起動可能なリストとして共存表示される（計算結果表示によってファイル検索結果が置き換わったり非表示になったりしない）
  - 計算結果表示欄は URL エンコード/デコード結果と同じ位置づけで、↑↓キーの選択対象に含まれる（選択中に Enter またはクリックで結果をクリップボードにコピーしてウィンドウを閉じる。コピー時にカンマ区切りを含めるかは設定画面で切替可能。デフォルトはカンマあり）
  - 数式には括弧 `( )` を使用でき、優先順位を明示した計算が可能（例: `(1+3)/3`、`2*(3+4)`）
  - 数式のパースに失敗した場合（閉じ括弧の不足など）は計算結果表示欄を表示せず、ファイル検索結果のみを表示する
  - 数式計算と URL エンコード/デコードが同一入力で同時に成立することは、入力の文字種上構造的に発生しない。数式らしい入力の判定（`isCalcExpression`）は数字・演算子・括弧・空白・小数点のみを許容しレターを含められないのに対し、URL エンコード/デコードの判定（`isUrlLikeInput`）は入力全体が `http://` または `https://` で始まる、すなわちレターを含むことを前提とするため、両者の判定条件は互いに排他になる
  - 万一両方の固定表示欄が発生し得る場合に備え、表示・選択インデックスの優先順位は「計算結果（インデックス0）→ URL エンコード/デコード結果（その直後）→ ファイル検索結果」の順に固定する
- 検索ボックスの入力内容に応じて URL エンコード/デコード結果を検索結果の一つとして自動表示し、Enter で結果をクリップボードにコピーしてウィンドウを閉じる（詳細は「計算・変換」節内の「URLエンコード/デコードの自動表示」を参照）
- Explorer でファイル/フォルダをコピー（Ctrl+C）した状態で検索ボックスに貼り付ける（Ctrl+V）と、数式計算・URLエンコード/デコードと同じ「暗黙判定」方式で検索フォルダの追加・ショートカットの配置を行える（詳細は「パス貼り付けによる検索フォルダ管理」節を参照）
- 検索対象フォルダは設定画面で追加・削除・有効/無効の切替が可能（複数フォルダ対応）
- 設定はアプリ終了後も永続化する
- システムコマンドに対応する（固定の区切り文字 `/` に続けて、コマンドごとに設定画面で個別に変更可能なキーワード（デフォルト：`shutdown`・`restart`・`sleep`）を入力すると候補として表示され、例: `/shutdown`、Enter で確認ダイアログを表示し、再度確認後に実行）。`/` から始まらない入力、または `/` に続く文字列がどのキーワードにも前方一致しない入力（プレフィックスなしで「shutdown」等とだけ入力した場合を含む）は候補を表示せず、通常のファイル検索・Web検索の対象として扱う。システムコマンド候補が表示されている間はファイル検索（`search_files`）を呼び出さず、ファイル検索結果・計算結果・URL エンコード/デコード結果は表示しない（システムコマンド候補のみを排他的に表示する。詳細は「システムコマンド機能」節を参照）
- 検索ボックスに1文字以上入力されている場合、検索結果リストの末尾に「Googleで〇〇を検索」の固定行を常に表示する（〇〇は現在の入力キーワード）。クリック、またはキーボードカーソルで選択して Enter を押すとデフォルトブラウザで `https://www.google.com/search?q=〇〇` を開く
- ファイル検索／数式計算／URLエンコード・デコード／システムコマンド／Web検索／クリップボード履歴／パス貼り付けによる検索フォルダ管理の各機能は設定画面で個別に ON/OFF できる（デフォルトは全て ON）。OFF にした機能は検索ボックスの入力に反応しない

### ファイル検索結果のランキング（frecency）

- ファイルを起動するたびに、そのファイルの起動回数と最終起動時刻を記録する
- 検索結果は frecency スコア（起動回数と最終起動時刻の組み合わせ）の降順で並び替えて表示する。未起動のファイルはスコア0として扱い、アルファベット順で末尾に表示する
- frecency スコアの計算式：`score = count * decay`。decay は最終起動時刻からの経過時間に応じて減衰する係数（1時間以内=1.0、1日以内=0.9、1週間以内=0.7、1ヶ月以内=0.5、それ以上=0.3）
- 履歴データは永続化し、アプリ起動時に読み込む
- この機能は常に ON とし、設定画面での ON/OFF は不要

### URLエンコード/デコードの自動表示

- 設定画面上は「計算・変換」カテゴリ配下のサブ機能として位置づけ、数式計算（「基本動作」節を参照）と並ぶ機能ブロックとして扱う（詳細は「設定画面」節の「計算・変換」を参照）
- 検索ボックスへの入力内容に応じて、URLエンコード/デコードの結果を検索結果の一つとして自動表示する。数式計算機能と同様に、検索結果の一部として扱う（ファイル検索結果など他の検索結果を置き換えるものではなく、共存させる）
- 前提条件：入力全体が `http://` または `https://` で始まる場合のみ、以下の判定ロジック（デコード／エンコード）を実行し、結果を表示する
  - この条件を満たさない場合は、`%XX` パターンや非ASCII文字を含んでいても、エンコード/デコード結果は一切表示しない（通常のファイル検索結果・アプリ起動結果などのみを表示する）
  - 理由：`http(s)://` で始まらない通常の日本語検索（例:「ありがとう」「家計簿」）でも、非ASCII文字を含むというだけの理由でエンコード結果が表示され、本来優先されるべきアプリ起動結果などが押し下げられてノイズになっていたため。URLとして明確に判別できる入力のみに対象を限定することでこれを解消する
- 判定ロジックは、上記の前提条件を満たした入力に対して、以下の優先順で判定する
  1. 入力に `%XX`（`%` に続く16進数2桁）のパターンが含まれる場合、URLデコードを実行する。デコード結果が有効なUTF-8文字列に変換でき、かつ元の入力と異なる結果が得られた場合のみ、デコード結果を検索結果として表示する
     - `%` の直後が16進数2桁でない場合（例:「100% off」の `% o`）は `%XX` パターンとして扱わず、デコード候補と誤判定しない
     - デコード結果中のスペースの扱いは「スペースは%20のまま保持する」設定（デフォルト OFF）に従う。OFF の場合はスペース（`%20`）を含め通常通りすべてデコードして半角スペースとして表示する。ON の場合、`%20` のみデコードせずそのまま表示に残し、それ以外の `%XX`（`%2B` 等）は従来通りすべてデコードする。この設定はデコード処理のみに影響し、後述のエンコード処理には影響しない
     - 「元の入力と異なる結果が得られた場合のみ表示する」の判定（無変化フィルタ）は、常に「スペース保持設定を無視した完全デコード結果」（＝ `%20` も含めすべての `%XX` をデコードした場合の結果）と入力を比較して行う。表示する文字列自体（実際に画面に出す値）は「スペース保持設定を反映した結果」を使う
       - この判定基準を採用する理由：無変化フィルタを「スペース保持設定を反映した後の文字列」と入力の比較で行うと、`%20` 以外にデコード対象がない入力（例: `https://test.com/サイト%20OK/test.html`）で「スペースは%20のまま保持する」が ON の場合、表示用文字列が入力と完全に一致してしまい、実際にはデコード対象（`%20`）が存在するにもかかわらず結果が非表示になってしまう。完全デコード結果との比較に基準を統一することで、この誤判定を避ける
       - 上記の例のように、無変化フィルタは「異なる」と判定するが、スペース保持設定により表示文字列が入力と見た目上一致することもある。これは正しい挙動であり、後述のラベル表示（「デコード結果」）によって計算結果であることを利用者が識別できるようにする
  2. 上記1に該当せず、入力に非ASCII文字（日本語などのマルチバイト文字）が含まれる場合、URLエンコードを実行し、その結果を検索結果として表示する
     - エンコードは `encodeURIComponent` 相当ではなく `encodeURI` 相当の挙動とする。`:` `/` `?` `#` `[` `]` `@` `!` `$` `&` `'` `(` `)` `*` `+` `,` `;` `=` など、URLの構造を保つ記号はエンコードせず、非ASCII文字のみをパーセントエンコードする
     - 例: 入力 `https://claude.ai/chat/お疲れ様！` → `https://claude.ai/chat/%E3%81%8A%E7%96%B2%E3%82%8C%E6%A7%98%EF%BC%81` のように、URL構造部分はそのまま残す
  3. 上記のどちらにも該当しない場合（英数字のみで、有効な `%XX` パターンも含まない場合）は、追加の検索結果を表示しない
- 表示形式は数式計算結果と同様に、検索結果リストの先頭付近に表示する
  - 結果本文の上に、処理種別を示す小さなラベル（デコード結果を表示する場合は「デコード結果」、エンコード結果を表示する場合は「エンコード結果」）を、結果本文より小さく控えめな見た目（グレーの小文字）で表示する。既存の検索結果アイテムの補足テキスト（ファイルパスや「Enter でコピー」等）と同系統のスタイルを踏襲する
- Enter またはクリックで結果をクリップボードにコピーしてウィンドウを閉じる
- 設定画面の「計算・変換」カテゴリ内に、「URLエンコード/デコードの自動表示」の ON/OFF を切り替えるチェックボックスを設ける
- 既存の数式計算機能と同様、このカテゴリ配下の独立したサブ項目として扱う
- 主設定チェックボックスに従属する設定として、「スペースは%20のまま保持する」チェックボックスを設ける（デフォルト OFF）
  - OFF（デフォルト）の場合、デコード結果のスペースは半角スペースとして表示される（現状の挙動を維持）
  - ON にすると、スペースだけは `%20` のまま保持される（スペースをURLの終端と誤認識するアプリ対策）
  - 適用範囲はデコード処理のみで、エンコード処理（非ASCII文字入力時のURI変換）には影響しない
  - 対象は半角スペース（`%20`）のみとし、`%2B` 等の他の記号は設定値に関わらず従来通りすべてデコードする

### パス貼り付けによる検索フォルダ管理

#### 機能概要

- Explorer でファイル/フォルダをコピー（Ctrl+C）した状態で WinLauncher を開き、検索ボックスに貼り付ける（Ctrl+V）と、既存の数式計算・URLエンコード/デコードと同じ「暗黙判定」方式（入力内容から自動的にモードを判定し、明示プレフィックスを必要としない方式）で、以下の2つの操作を行える
  1. 検索フォルダとして追加（対象: フォルダのみ）
  2. 検索フォルダにショートカットとして追加（対象: フォルダ・ファイル両方）
- 設定画面のカテゴリは新設の「パス貼り付け」とする（詳細は「設定画面」節を参照）

#### 貼り付け内容の判定方法

1. Ctrl+V による貼り付け操作を検知した際、クリップボードに `CF_HDROP` 形式（Explorer での通常コピー時に付与される、ファイル/フォルダの実体パス一覧）が存在するか確認する
2. `CF_HDROP` が存在し、かつパスが1件のみの場合、そのパスの文字列を検索ボックスに直接流し込む（クォート等の加工は行わず、生のパス文字列のまま）。これにより後続の判定処理を、通常のテキスト貼り付け（パスのコピー等）と完全に共通化する
3. `CF_HDROP` に複数パスが含まれる場合（複数選択してコピーした場合）は、検索ボックスへの流し込みを行わない（通常の Ctrl+V と同じ空振り状態にする。暗黙判定も発火しない）
4. `CF_HDROP` が存在しない場合は、通常の Ctrl+V（OS標準のペースト処理）に任せる（テキストとしてクリップボードの内容がそのまま検索ボックスに入る、既存の挙動）
5. 検索ボックスに入力された文字列（上記2、または通常のペースト・手入力いずれの経路でも）に対して、実在するパスかどうかを判定する処理を行う。この判定はダブルクォートの有無に関わらず動作する（前後のダブルクォートのみを取り除いてから判定するため、「パスのコピー」でクリップボードに入る `"C:\Users\...\file.txt"` のようなダブルクォート付き文字列、クォートなしの生のパス文字列のいずれも同様に判定できる）
6. 判定したパスが実在しない場合は、通常の検索文字列として扱う（暗黙判定を発火させない）

#### 表示順序

- 判定したパスがフォルダの場合、候補行を上から以下の順で表示する
  1. 検索フォルダにショートカットとして追加（機能2）
  2. 検索フォルダとして追加（機能1）
  3. ファイル検索結果（共存する場合。「モード共存・排他一覧」節を参照）
  4. 「Googleで〇〇を検索」行（Web検索）
- 判定したパスがファイルの場合（機能1はフォルダ限定のため対象外）、候補行を上から以下の順で表示する
  1. 検索フォルダにショートカットとして追加（機能2）
  2. ファイル検索結果（共存する場合）
  3. 「Googleで〇〇を検索」行（Web検索）
- 数式計算結果・URLエンコード/デコード結果との関係：貼り付け判定で得られるパス文字列（ローカルファイルパス）は、数式計算の判定条件（数字・演算子・括弧・空白・小数点のみ）にも URL エンコード/デコードの判定条件（入力全体が `http://` または `https://` で始まる）にも該当しない文字構成のため、これらと同一入力で同時に成立することは構造的に発生しない。したがって「基本動作」節で定めた固定順序（計算結果 → URL エンコード/デコード結果 → ファイル検索結果）との優先関係を別途定める必要はない

#### 機能1: 検索フォルダとして追加（対象: フォルダのみ）

- 判定したパスがフォルダの場合、上記の表示順序に従い「検索フォルダに追加: <フォルダ名>」という候補行を表示する
- Enter で確定すると、「ウィンドウを閉じる系アクションの共通設計」節の設計原則（`hideWindow()` を最優先で待ち、フォルダ登録処理は非表示解決後の後処理として行う）に乗せて検索フォルダとして登録する
- 登録完了後、Windows のトースト通知で「検索フォルダに追加しました: <フォルダ名>」を表示する
- 既に登録済みのフォルダだった場合も、同様にトースト通知で「既に登録済みです」と伝え、エラー扱いにはしない

#### 機能2: 検索フォルダにショートカットとして追加（対象: フォルダ・ファイル両方）

- 判定したパスがフォルダまたはファイルの場合、上記の表示順序に従い「検索フォルダにショートカット配置: <名前>」という候補行を表示する
- Enter を押すと、以下の3ステップから成るミニウィザードに遷移する。既存の暗黙判定・プレフィックスコマンドが採る「Enter一発で確定・実行」パターン（数式計算・URLエンコード/デコード・システムコマンドの確認ダイアログ等）とは異なる、複数ステップの入力を伴う新しいインタラクションパターンとして扱う
  1. **パス貼り付け直後**：上記の候補行を表示する
  2. **ステップ2（フォルダ選択）**：候補行で Enter を押すと、登録済みの検索フォルダ一覧を選択肢として表示する（「プレフィックスコマンド候補表示」節と同様のリストUIを流用する）
  3. **ステップ3（名前編集）**：フォルダを選択すると、ショートカット名の編集欄を表示する（デフォルト値は元のファイル/フォルダ名）。編集して Enter を押すと保存を実行する
- ウィザード進行中（ステップ2・ステップ3）は、通常のファイル検索・他の暗黙判定モードとは排他的に振る舞う（ウィザードの選択肢・入力欄のみを表示する。詳細は「モード共存・排他一覧」節を参照）
- Escape キーは、ウィザードを1ステップ前に戻す（名前編集中→フォルダ選択に戻る、フォルダ選択中→通常の検索状態（候補行表示、またはファイル検索結果表示）に戻る）。既存の Esc＝ウィンドウ非表示という基本挙動の例外として扱う（詳細は「キー操作」節を参照）
- 保存（ステップ3の Enter）は、機能1と同じ「ウィンドウを閉じる系アクションの共通設計」に乗せ、`.lnk` ファイルの作成を非表示後に裏で行う
- 保存完了後、Windows のトースト通知で「ショートカットを配置しました: <名前>」を表示する
- 同名の `.lnk` が既に存在する場合は、Explorer 標準の挙動に倣い「名前 (2).lnk」のように自動的に連番を付与する（上書き・エラーにはしない）

#### 設定

- 本機能全体（機能1・機能2をまとめて）の ON/OFF を設定画面の「パス貼り付け」カテゴリに設ける（デフォルト ON。詳細は「設定画面」節を参照）
- OFF の場合、`CF_HDROP` 形式・テキスト形式いずれのパス貼り付けであっても暗黙判定は発火せず、通常の検索文字列として扱われる

### システムコマンド機能

#### プレフィックスでの呼び出し

- 明示プレフィックスは「`/` + キーワード」の2部構成とする
  - 先頭の `/` は固定の区切り文字であり、ユーザーは変更できない（設定画面から `/` 自体を変更する項目は設けない）
  - `/` に続くキーワード部分は、コマンドごとに設定画面で個別に変更可能（デフォルト：`shutdown`・`restart`・`sleep`）
- `/` に続けてキーワードを入力すると、該当するシステムコマンドの候補が前方一致で表示される（例: `/shutdown`）
- キーワードは設定画面の「システムコマンド」カテゴリで、3つの独立したテキスト入力としてそれぞれ変更できる（1つの共通プレフィックス設定ではない）
- `/` から始まらない入力、または `/` に続く文字列がどのキーワードにも前方一致しない入力（プレフィックスなしで「shutdown」等とだけ入力した場合を含む）は候補を表示せず、通常のファイル検索・Web検索の対象として扱う
- ファイル検索結果との関係は排他とする（候補表示中は `search_files` を呼び出さず、ファイル検索結果・計算結果・URLエンコード/デコード結果は表示しない）。Windowsのファイル名に `/` を使用できないため、共存させる実益がなく排他のままとする

#### 対応コマンド

- `/` + キーワード（デフォルト：`shutdown`） → シャットダウン
- `/` + キーワード（デフォルト：`restart`） → 再起動（従来の `reboot`（`restart` と同一アクション）は廃止し、`restart` に一本化する）
- `/` + キーワード（デフォルト：`sleep`） → スリープ
- `/` に続く文字列と各コマンドのキーワードとの前方一致で候補を絞り込む（例: キーワードがデフォルトの `restart` のままなら `/re` → `/restart` のみに絞り込み。キーワードを `sd` に変更した場合は `/sd` で候補が表示される）

#### 実行

- 候補を Enter またはクリックすると即実行せず、確認ダイアログを表示する
- 確認ダイアログで再度確認（実行操作）した場合にのみ、実際にコマンドを実行する
- 実行内容自体は変更しない

### クリップボード履歴

#### 呼び出し

- 明示プレフィックスは「`/` + 呼び出しキーワード」の2部構成とする
  - 先頭の `/` は固定の区切り文字であり、ユーザーは変更できない
  - `/` に続く呼び出しキーワード部分（デフォルト：`cb`）は設定画面で変更可能
- 検索ボックスに `/` ＋ 設定済みの呼び出しキーワードを入力するとクリップボード履歴モードに切り替わり、履歴一覧を表示する（例: `/cb`）
- 呼び出しキーワードは設定画面の「クリップボード」カテゴリの「呼び出しキーワード」欄で変更可能。`/` が自動的に先頭に付与される旨を画面上に明記する
- 呼び出しキーワードの後ろに続く文字列は、履歴一覧に対する絞り込みフィルタとして使う（既存の挙動を変更しない）

#### 履歴の記録

- アプリがバックグラウンドにいる間も、Windows の `AddClipboardFormatListener` API（イベント駆動）でクリップボードの変化を監視し、テキスト・画像を自動記録する
- 重複は除外する（同じ内容は最新の1件のみ保持し、先頭に移動する）
- 最大保持件数はデフォルト50件。設定画面で変更可能（上限を超えた古い履歴は削除する）。入力値の妥当性チェックの詳細は「クリップボード」設定カテゴリの節を参照
- 履歴データは永続化する。ただし画像データはメモリ上のみで保持し、アプリ再起動後は失われる（テキストのみ永続化）
- 画像データは Rust 側のメモリにバイナリのままキャッシュし、フロントエンドには画像ID・縮小サムネイル・サイズのみを渡す（巨大な画像データをフロントエンドとの間でやり取りしない。大きな画像でコピー操作が重くなる問題への対処）
- 履歴から画像を選択してコピーする際は、画像IDを指定してクリップボードへの書き戻しを Rust 側に依頼し、Rust が直接 Windows API でクリップボードに書き込む（フロントエンド側でのデコード処理は行わない）

#### 履歴一覧の表示

- 最新のものから順に左リストに表示する
- テキスト：先頭の数十文字をリストに表示する
- 画像：画像アイコン＋コピー日時をリストに表示する
- 上下キーで選択、Enter またはクリックで選択した内容をクリップボードにセットしてウィンドウを閉じる
- ウィンドウを閉じる際、検索ボックスのクエリはプレフィックス部分（`/` ＋ 呼び出しキーワード。例: `/cb`）のみを残し、それに続く絞り込みフィルタ文字列だけをクリアする（例: `/cbメモ` と絞り込んで選択した場合、次回ウィンドウを開いた際のクエリは `/cb` に戻る）。ファイル検索など他のモードで確定時にクエリを完全に空文字へ戻す挙動とは異なる、クリップボード履歴・最近使ったファイル一覧固有の挙動である

#### ペイン分割のリサイズ

- 左リスト（履歴一覧）と右パネル（プレビュー）の境界線をマウスでドラッグして分割幅を変更できる
- ドラッグ中はカーソルを col-resize に変更する
- 左ペインの最小幅は 150px、最大幅はウィンドウ幅の 60% とする
- 分割幅は tauri-plugin-store で永続化し、アプリ再起動後・フォーカスアウト後も保持する

#### 右パネル（クリップボード履歴モード時のみ表示）

- テキスト：本文をある程度表示し、下部にコピー日時・文字数等のメタ情報を表示する
- 画像：サムネイルを表示し、下部にコピー日時・サイズ等のメタ情報を表示する

### 最近使ったファイル一覧（/recent）

#### 呼び出し（最近使ったファイル一覧）

- 明示プレフィックスは「`/` + 呼び出しキーワード」の2部構成とする
  - 先頭の `/` は固定の区切り文字であり、ユーザーは変更できない
  - `/` に続く呼び出しキーワード部分（デフォルト：`recent`）は設定画面で変更可能
- 検索ボックスに `/` ＋ 設定済みの呼び出しキーワードを入力すると最近使ったファイル一覧モードに切り替わり、一覧を表示する（例: `/recent`）
- 呼び出しキーワードは設定画面の「最近使ったファイル」カテゴリの「呼び出しキーワード」欄で変更可能。`/` が自動的に先頭に付与される旨を画面上に明記する
- 呼び出しキーワードの後ろに続く文字列は、一覧に対する絞り込みフィルタとして使う（「フィルタ」を参照）
- 他のプレフィックスコマンドと同様、変更しようとしたキーワードが既存の他キーワードと重複する場合は保存を拒否する（「プレフィックスキーワードの重複チェック」節を参照）
- ファイル検索結果との関係は他のプレフィックスコマンドと同様に排他とする（一覧表示中は `search_files` を呼び出さず、ファイル検索結果・計算結果・URLエンコード/デコード結果は表示しない）

#### データソース

- 以下の2つのフォルダを対象とする
  1. Windows の Recent フォルダ（`%APPDATA%\Microsoft\Windows\Recent`。最近使ったファイルへのショートカットが保存される場所）。パスはハードコードせず、Windows の Known Folder API（`FOLDERID_Recent`）で取得する（環境によって実際のパスが異なり得るため）
  2. Office の Recent フォルダ（`%APPDATA%\Microsoft\Office\Recent`。Office アプリで最近使ったファイルへの参照が保存される場所）。対応する Known Folder API が存在しないため、`%APPDATA%` のパスを取得したうえで `\Microsoft\Office\Recent` を結合して構築する

#### ファイル種別ごとの扱い

両フォルダには `.lnk`（ショートカット）と `.url`（インターネットショートカット）の2種類のファイルが混在し得る。それぞれ以下の方式で扱う。

##### .lnk（ショートカット。両フォルダに存在しうる）

- 既存仕様通り、リンク先のローカルパスを解決する
- リンク先がフォルダである場合、または既に削除され実在しない場合は一覧から除外する（ネットワークパス（UNC）の場合の実在チェックの扱いは「実在チェックとネットワークパス（UNC）の扱い」節を参照）
- ソートキーは `.lnk` ファイル自体の更新日時（mtime）とする（リンク先実ファイルのタイムスタンプではない。理由は「ソート順」節を参照）
- Windows の Recent フォルダと Office の Recent フォルダの両方に、同一のリンク先ローカルパスを指す `.lnk` が存在する場合は1件に統合する（`.lnk` 自体の更新日時が新しい方を採用する）
- 日本語ファイル名や日本語を含むパスを持つ `.lnk` が一覧から欠落しないよう、`.lnk` のパース時はシステムの既定 ANSI コードページ（`GetACP()`）に対応する文字コードでデコードする（固定で単一の欧文コードページを使うと、Shift-JIS 等のパスがデコード不能となり静かに欠落する）

##### .url（インターネットショートカット。主に Office の Recent フォルダに存在し、OneDrive・SharePoint 等クラウド上の Office ファイルを指す）

- テキスト（INI形式）としてパースし、`URL=` で始まる行の値を URL 文字列として取得する
- 取得した URL 文字列を同期ライブラリのローカルパスへ変換したうえで一覧に含める。変換手順は以下の通り
  1. レジストリ `HKEY_CURRENT_USER\Software\SyncEngines\Providers\OneDrive` 配下のすべてのサブキーを列挙し、各サブキーの `UrlNamespace`（そのライブラリのクラウド上URLのルート）と `MountPoint`（対応するローカル同期先フォルダのパス）の組を「クラウドURLプレフィックス → ローカルパス」の対応表として保持する。実地検証の結果、個人のOneDrive本体・OneDrive for Businessの個人領域・SharePointチームサイトの共有ライブラリ・OneDriveに追加したショートカットのいずれも、この同じレジストリ配下に登録されることが確認されている（例: 個人アカウントの場合 `UrlNamespace=https://d.docs.live.net`、`MountPoint=D:\OneDrive` のように登録される）
     - レジストリ値の型が `REG_EXPAND_SZ` の場合、`MountPoint` が環境変数のプレースホルダー（例: `%UserProfile%\OneDrive`）を含んだまま展開されずに返されることが実環境での検証で確認されている。取得した値にプレースホルダーが含まれる場合は展開してからパスとして使用する
  2. URL 文字列に対し、対応表の中から `UrlNamespace` が前方一致（prefix）するエントリを探す。該当するエントリが複数ある場合は、最も長く一致する `UrlNamespace` を採用する（最長一致優先）
  3. 一致したら、URLから `UrlNamespace` 部分を除いた残りを相対パスとし、パーセントエンコーディング（`%20` 等）をデコードしたうえで、対応する `MountPoint` と結合してローカルパスを組み立てる
  4. 組み立てたローカルパスの実在確認を行う（ネットワークパス（UNC）の場合の扱いは「実在チェックとネットワークパス（UNC）の扱い」節を参照）。存在すれば採用する
- 対応表の中に `UrlNamespace` が前方一致するエントリが見つからない場合、またはローカルパスが実在しない場合は、削除済みファイルと同じ扱いとして一覧から除外する
- 従来のレジストリ `HKEY_CURRENT_USER\Software\Microsoft\OneDrive\Accounts` 配下の `UserFolder` 値を使った変換方式（個人OneDriveの `d.docs.live.net` 形式URLのみに対応していた旧方式）は、実地検証の結果、上記の `SyncEngines\Providers\OneDrive` ベースの新方式のみで完全に代替できることが確認されたため、フォールバックとしても残さず廃止する
- 既知の制約：ライブラリ全体ではなく一部のサブフォルダのみを同期している場合、`UrlNamespace` と `MountPoint` の対応関係にズレが生じ、正しく変換できないケースがあることが分かっている。発生頻度は低いと見込まれるため、今回のスコープでは許容する既知の制約として記載するに留める
- ローカルパスへの変換に成功したエントリは、以降 `.lnk` 由来のエントリと全く同じ扱い（実在確認済みのローカルファイルとして表示・起動）にする。同一のローカルパスを指す `.lnk` エントリが存在する場合の重複統合（mtime が新しい方を採用。「.lnk」節を参照）の対象にも含める
- 表示名は「`.url` ファイル名から末尾の拡張子 `.url` を取り除いたもの」とする（例: ファイル名が `教育論_FY2026.xlsx.url` の場合、表示名は `教育論_FY2026.xlsx`。Windows のエクスプローラーが `.url` の拡張子を隠して表示するのと同じ見た目にするため）。上記の処理で拡張子を取り除いた結果、表示名に拡張子が残らないもの（例: `d.docs.live.net の 20260402_教育論` のような、OneDrive 上のフォルダ的な参照）は、「ファイルのみを対象とし、フォルダは除外する」という既存ルールに従い一覧から除外する
- ソートキーは `.url` ファイル自体の更新日時（mtime）とする（ローカルパスへの変換の成否に関わらず変わらない）

#### 実在チェックとネットワークパス（UNC）の扱い

- リンク先のパス（`.lnk` 由来・`.url` 変換後いずれも）が `\\` で始まるUNC形式（ネットワークパス）である場合、実在チェックそのものを行わず、無条件で一覧に含める（削除済みファイル除外の対象外とする）
  - 背景：`.lnk`・変換後の `.url` いずれかのリンク先がネットワークドライブ/共有フォルダ等のネットワークパスである場合、実在チェックがネットワーク越しの応答待ちとなり、応答が遅い・到達不能な対象が数件混ざるだけで一覧表示全体が大幅に遅延する不具合が実環境での検証で確認されている（実地検証では一覧表示に約15秒かかる事例を確認）
  - この結果、既にサーバー上から削除されたファイルがまれに一覧に残り、選択時に「ファイルが見つかりません」となる可能性があるが、許容する既知の制約とする
- ローカルパス（`C:\` 等のドライブレター形式）については、既存通り実在チェックを行う

#### フィルタ

- `/recent` に続けて文字列を入力すると、既存のファイル検索と同様にインクリメンタルに絞り込む
- 絞り込みは、入力文字列を各エントリの表示名（`.lnk` はリンク先ファイル名そのもの、`.url` は拡張子除去後の名称。「ファイル種別ごとの扱い」節を参照）に対する部分一致でフィルタする単一ロジックとする。拡張子専用の判定ロジックや既知拡張子リストは持たない
- 表示名文字列には（`.url` 由来のフォルダ的参照を除き）拡張子がそのまま含まれるため、この部分一致だけで拡張子的な絞り込み（例: `xls` と入力すると `.xlsx` と `.xls` の両方がヒットする）も副次的に成立する
- `.lnk` 由来・`.url` 由来を問わず絞り込みロジックは共通とする（種別による特別扱いは不要）

#### ソート順

- 一覧は常に「最終アクセス日時が新しい順」でソートする。絞り込み後も同じ順序を維持する（frecency によるスコア並び替えは行わない）
- 「最終アクセス日時」は各エントリの由来ファイル（`.lnk` または `.url`）自体の更新日時を指す。両フォルダ・両ファイル種別のエントリを合わせて1つの一覧としてソートする

#### 表示件数

- ファイル検索結果・プレフィックスコマンド候補一覧とは独立した、最近使ったファイル一覧専用の表示件数上限を持つ（デフォルト：50件）
- 表示件数上限は設定画面の「最近使ったファイル」カテゴリの「最大表示件数」欄で変更可能。入力値の妥当性：1件以上200件以下の正の整数のみを許可する。範囲外・不正な値が入力された場合は保存せず、エラー表示とする

#### 期間による足切り

- 上記の件数上限に加え、最終アクセス日時が現在時刻から「保持期間」より前のエントリは、件数上限に余裕があるかどうかに関わらず一覧に表示しない（除外する）
- 足切り条件は「件数上限」と「保持期間以内」の両方であり、どちらか一方の条件を満たさなくなった時点で除外する
- 結果として、直近の保持期間内に開かれたファイルが件数上限に満たない場合、一覧の件数はそれより少なくなることがある（意図した挙動）
- 保持期間はデフォルト180日とし、設定画面の「最近使ったファイル」カテゴリの「保持期間」欄で変更可能。入力値の妥当性：1日以上3650日（10年）以下の正の整数のみを許可する。範囲外・不正な値が入力された場合は保存せず、エラー表示とする
- 保持期間のデフォルト値は将来的に調整しうる値のため、実装時はマジックナンバーとして埋め込まず、定数として分かりやすい場所にまとめて定義することが望ましい

#### 選択と起動

- 上下キーで選択、Enter またはクリックで起動する
- 一覧に表示されるエントリはすべてローカルパスを保持している（`.url` 由来のエントリも、ローカルパスへの変換に成功したものだけが一覧に含まれるため）。起動処理を由来によって分岐させる必要はなく、既存の「ファイル起動」の挙動と同様にデフォルトアプリで起動する
- ウィンドウを閉じる
- ウィンドウを閉じる際、検索ボックスのクエリはプレフィックス部分（`/` ＋ 呼び出しキーワード。例: `/recent`）のみを残し、それに続く絞り込みフィルタ文字列だけをクリアする（例: `/recentレポート` と絞り込んでファイルを起動した場合、次回ウィンドウを開いた際のクエリは `/recent` に戻る）。ファイル検索など他のモードで確定時にクエリを完全に空文字へ戻す挙動とは異なる、最近使ったファイル一覧・クリップボード履歴固有の挙動である

### 格納フォルダを開く（Shift+Enter）

#### 対象範囲

- 通常のファイル検索結果、および「最近使ったファイル一覧（/recent）」の結果一覧の両方を対象とする
- 計算結果・URLエンコード/デコード結果・システムコマンド候補・クリップボード履歴・プレフィックスコマンド候補は対象外とする（いずれもファイルパスを持たないため）
- パス貼り付けによる検索フォルダ管理の候補行（検索フォルダに追加／検索フォルダにショートカット配置）、およびそのウィザード中の選択肢・入力欄も対象外とする（「パス貼り付けによる検索フォルダ管理」節を参照）

#### 挙動

- 選択中の項目に対して Shift+Enter を押すと、そのファイルの格納フォルダ（親フォルダ）をエクスプローラーで開く
- 対象ファイル自体の選択・ハイライトは行わない（シンプルにフォルダを開くのみ）
- エクスプローラーが前面に出ると WinLauncher のウィンドウが背後に残り邪魔になるため、通常の Enter によるファイル起動と同様にウィンドウを閉じる（非表示にする）。フォーカスアウトによる自動非表示（150msのデバウンス処理）に委ねるのではなく、通常のファイル起動時と同じ体感速度で閉じるため、フォルダを開く処理の直後に明示的にウィンドウを非表示にする処理を呼び出す

#### .lnk（ショートカット）の扱い

- 通常のファイル検索結果に `.lnk`（ショートカット）自体が表示されているケースでは、「`.lnk` 自身が置かれているフォルダ」ではなく「`.lnk` が指すリンク先の実ファイルが置かれているフォルダ」を開く
  - 「最近使ったファイル一覧（/recent）」節の `.lnk` の扱い（リンク先のローカルパスを解決するロジック。文字コード処理・panic対策を含む）と同じリンク解決ロジックを再利用し、通常のファイル検索側・`/recent` 側の双方で同一の挙動に揃える
  - リンク解決に失敗した場合（リンク先が実在しない等）は、フォールバックとして `.lnk` 自身が置かれているフォルダを開く
- `.url`（インターネットショートカット）は通常のファイル検索結果には出現しない（`/recent` 固有の取得対象であり、`/recent` 側は取得時点でリンク先の実パスに解決済みのため、本機能では通常ファイル検索側の `.url` 対応は考慮不要）

#### ON/OFF

- 機能 ON/OFF のトグルは設けない。Ctrl+S・Esc 等、既存の常時有効なキーボードショートカットと同様の扱いとする

### プレフィックスキーワードの重複チェック

- システムコマンドの3キーワード（デフォルト：`shutdown`・`restart`・`sleep`）とクリップボード履歴の呼び出しキーワード（デフォルト：`cb`）、最近使ったファイル一覧の呼び出しキーワード（デフォルト：`recent`）、および今後追加される全プレフィックス機能のキーワードは、互いに重複してはならない
- いずれかのキーワードを設定画面で変更する際、他の既存キーワード（自分自身を除く）と完全一致する場合は保存を拒否し、エラーとして扱う。設定画面にエラーメッセージを表示し、変更前の値を維持する
- 大文字・小文字の区別は、既存のプレフィックス判定ロジック（`/` に続くキーワード部分の前方一致判定）における大文字・小文字の区別の有無に合わせる
- キーワードが空文字列の場合は、重複チェック以前の問題として保存を拒否する（既存の空文字列エラーと同様の扱い）

### プレフィックスコマンド候補表示

- 検索ボックスの入力が `/` から始まる場合、登録済みの全コマンド（デフォルトでは `/shutdown`、`/restart`、`/sleep`、`/cb`、`/recent`。各キーワードは設定画面で変更可能。詳細は「システムコマンド機能」「クリップボード履歴」「最近使ったファイル一覧（/recent）」の各節を参照）を候補として、ファイル検索結果とは別枠の一覧表示領域に表示する
- `/` 単体の入力時点で全プレフィックスコマンドを一覧表示する。それ以降に続く文字列で前方一致するものに絞り込む（例: `/re` → `/restart` のみに絞り込み）
- 各候補には、コマンド名（例: `/restart`）とその説明文（例:「再起動」）を表示する
- 候補は使用頻度に応じて並び替える。ファイル検索結果の frecency ランキング（「ファイル検索結果のランキング（frecency）」節を参照）と同じ考え方で、使用回数と直近使用日時に基づくスコアの降順でソートする。使用実績のないコマンドはスコア0として扱い、その場合はコマンド名のアルファベット順を二次キーとする
  - 使用実績（使用回数・直近使用日時）はプレフィックスコマンドの実行都度記録し、tauri-plugin-store で永続化する（ファイル検索の frecency と同様の方式）
- 候補一覧は上下キーで選択でき、選択中に Enter またはクリックで直接そのコマンドを実行する（ファイル検索結果の選択・実行と同じ挙動）。システムコマンドの場合は既存の確認ダイアログを経由する
- プレフィックスに完全一致しない入力（前方一致する候補が0件の場合）は候補欄を表示せず、通常のファイル検索結果を表示する

### OCR機能（画像からのテキスト抽出）

- 検索ボックスにフォーカスがある状態で画像をペースト（Ctrl+V）すると、画像内のテキストをOCRで抽出する
- OCRエンジンは Windows OCR API（Windows.Media.Ocr）を使用する。Rust側から windows-rs クレート経由で呼び出す
- OS側に日本語のOCR言語パックが導入されていることを前提に、Windows OCR APIでテキストを抽出する（対応言語が不足する場合は、設定→時刻と言語→言語からOCR言語パックを追加する必要がある旨、READMEまたは注記に明記する）
- 抽出結果は検索ボックスには反映せず、検索ボックスの下に専用のプレビュー領域を表示する
  - プレビュー領域のテキストは編集可能とする
  - 認識された改行はそのまま保持し、元画像のレイアウトに近い形で表示する
  - プレビュー領域にコピーボタンを設置し、クリックで表示中のテキスト全体をクリップボードにコピーする
- OCR処理中はローディング状態を表示する（処理に数百ms〜数秒かかるため）
- 本機能は既存のクリップボード履歴機能（クリップボードの変化を監視して記録する仕組み）とは独立した機能であり、相互に影響しない
- 設定画面にON/OFFトグルを追加する（デフォルトはON）。カテゴリは既存の「クリップボード」内、または新規「OCR」カテゴリのいずれかで検討し、実装しやすい方を選択する
- OCRプレビューが表示されている間、検索結果リスト（ファイル検索・計算結果・システムコマンド候補・Web検索候補等）は非表示にする
- OCRプレビューを閉じる操作は「閉じる」ボタンと「コピーして閉じる」ボタンの2種類があり、それぞれ以下の挙動とする
  - 「閉じる」ボタン：OCRプレビューを閉じ、検索結果リストの表示状態に戻したうえで、検索ボックス（input）にフォーカスを戻す
  - 「コピーして閉じる」ボタン：表示中のテキスト全体をクリップボードにコピーしたのち、ウィンドウ全体を150〜200ms程度でフェードアウトさせてから非表示にする（ホットキーやフォーカスアウトによる非表示と同様、ウィンドウの表示/非表示切り替えの一種として扱う）
  - Ctrl+D は、OCRプレビュー表示中に限り「閉じる」ボタンと同一の挙動を行うショートカットとする（詳細は「キー操作」節を参照）
  - いずれの操作でも、OCRプレビューの表示状態はリセットされ、次回ウィンドウを表示した際は通常の検索画面（OCRプレビューではない状態）に戻っている
- OCRプレビュー領域は、検索ボックスと同じ幅の枠内で、左右2ペインに分割して表示する
  - 左ペイン: 貼り付けられた画像のプレビュー（ウィンドウ・ペイン幅に応じて画像が縮小・拡大されるように表示する。アスペクト比は維持する）
  - 右ペイン: OCR結果のテキスト（編集可能なテキストエリア、コピー・閉じるボタンを含む、既存仕様通り）
- 左右ペインの境界線はマウスでドラッグして分割幅を変更できる（ドラッグ中はカーソルをcol-resizeに変更する）
- 左ペインの最小幅は150px、最大幅は全体幅の60%とする（クリップボード履歴のペイン分割リサイズと同じ制限）
- 分割幅は永続化しない。新しい画像が貼り付けられるたびに、分割幅は50:50の初期状態に戻る
- 高さは、検索ボックスの直下からウィンドウ下端まで全高を使う（既存仕様を維持）
- テキストエリア内の内容がプレビュー領域の高さを超える場合は、テキストエリア内でスクロール可能とする
- OCR結果の連結処理は、Windows OCR API の仕様上 OcrLine.Text は単語間に半角スペースを挿入した文字列であり、日本語・中国語等のCJK文字ではこれが不要な空白として現れるため、以下の方式で構築する
  - 各行（OcrLine）について、Words を個別に取得し、単語ごとに連結する
  - 連結時、直前の単語と現在の単語が両方ともASCII英数字（半角英字・半角数字）のみで構成されている場合に限り、間に半角スペースを1つ挿入する
  - それ以外の組み合わせ（日本語・記号・CJK文字を含む場合等）は、スペースを挿入せずそのまま連結する
- OCR結果の行の並び順は、認識順ではなく、各行の位置情報（バウンディングボックスの縦位置）を基準に上から下へ並べ替えてから連結し、視覚的なレイアウトと一致させる

### 自動アップデート機能

- Tauri v2 公式の `tauri-plugin-updater` を使用し、アプリ内から新バージョンの検出・ダウンロード・インストールを行う
- 配信方式は GitHub Releases ＋静的 `latest.json` 方式とし、既存のリリースフローと統合する

#### 起動時チェック

- アプリ起動時に自動でアップデートの有無をチェックする
- この挙動は設定画面でON/OFF切り替え可能とする（デフォルトはON）
- バックグラウンドでの定期的な自動チェックは行わない（起動時チェックのみで、アプリ起動中の再チェックはしない）

#### 手動チェック

- トレイメニューに「Check for Updates」項目を追加する（ラベルは英語固定）
- クリックすると起動時チェックと同じロジックでアップデートの有無を確認する

#### 更新確認〜インストールの流れ

- チェックの結果、新しいバージョンが見つかった場合、新バージョン番号とリリースノート（GitHub Releases の本文をそのまま、または要約して表示）をユーザーに簡易表示する
- ユーザーが明示的に同意（実行操作）した場合のみダウンロード＆インストールを行う。ユーザーの意図しないタイミングでの再起動を避けるため、サイレントでの自動更新は行わない
- Windows の制約上、インストール前にアプリは自動終了する。終了時（`on_before_exit` フック）にトレイアイコンの後片付けなど必要な終了処理を行う
- インストール完了後、アプリを自動的に再起動する

#### 署名

- Tauri Updater 独自の鍵ペア（`tauri signer generate` で生成、minisign方式）で更新パッケージに署名する
- これは Windows のコード署名（Authenticode）とは別物であり、今回のスコープには含めない
- 秘密鍵は GitHub Actions の Secrets 等、安全な場所で管理する

#### リリースフローへの影響

- 既存のリリース手順（`npm run tauri build` → `last-release-notes.md` 更新 → git commit/push/tag → `gh release create` → `gh release upload`）に、`latest.json` の生成・アップロードのステップが追加される

| Esc | ウィンドウを非表示 | 常時（設定画面表示中は設定画面を閉じて検索画面へ戻る） | クエリは保持される。パス貼り付けによる検索フォルダ管理のウィザード進行中はステップを1つ戻る（「パス貼り付けによる検索フォルダ管理」節を参照） |
| Ctrl+S | 設定画面の表示/非表示をトグル | 常時 | 検索ボックス右の設定ボタン（歯車アイコン）でも同様にトグル可能 |
| Shift+Enter | 選択中の項目の格納フォルダを開き、ウィンドウを閉じる | 通常ファイル検索・/recent のみ | 「格納フォルダを開く（Shift+Enter）」節を参照 |
| Ctrl+D | クエリを全クリア | 常時 | OCRプレビュー表示中のみ「閉じる」ボタンと同一の挙動（ウィンドウは閉じない）。「OCR機能（画像からのテキスト抽出）」節を参照 |

フッターのキー操作ヒント一覧には、上記表内の該当するキー操作を表示する。

フォーカスアウトで自動的にウィンドウを隠す（最後の検索キーワードは保持し、再表示時にそのキーワードが入力された状態にする）。これはキー操作ではなくウィンドウの自動非表示挙動のため、上記表には含めない。

### 設定画面

- 左側にカテゴリナビ（全般／ファイル検索／パス貼り付け／計算・変換／システムコマンド／Web検索／クリップボード／最近使ったファイル／このアプリについて）、右側に選択中カテゴリの設定内容を表示するタブ構成
- カテゴリナビの幅は、各カテゴリ名がどれも折り返さず1行で収まる幅を確保する（「最近使ったファイル」のような長いラベルで折り返しが発生しないこと）
- 設定内容はすべて永続化し、アプリ起動時に読み込んで反映する

#### 全般

- 起動ホットキーを変更できる。修飾キー（Ctrl / Alt / Shift / Win）をチェックボックスで複数選択し、通常キーを以下の順のプルダウンから選択する
  - Space / Tab / Enter / F1〜F12 / A〜Z / 0〜9
- 選択中の組み合わせをリアルタイムで「Alt+Space」のように表示する
- デフォルトは Alt+Space
- 修飾キー（Ctrl/Alt/Shift/Win）を1つ以上選択しないと保存できず、インラインでエラーを表示する
- 保存に失敗した場合（他アプリが使用中など）はエラーを表示し、変更前の設定を維持する
- 起動時の自動アップデートチェックのON/OFFトグル（デフォルトはON）

#### ファイル検索

- 機能 ON/OFF トグル（OFF 時は検索結果にファイルを一切出さない）
- 検索フォルダの追加・有効/無効切替・削除（既存機能をこのカテゴリ配下に配置）
- 検索フォルダ一覧の各フォルダパスのテキスト部分をクリックすると、Windows のエクスプローラーでそのフォルダを開く
- フォルダパスがクリック可能であることが視覚的にわかるよう、ホバー時にカーソルを pointer にするか下線を表示するなどの見た目上の変化を加える
- 有効/無効切り替え用チェックボックスや削除ボタンなどの既存の操作要素の挙動には影響を与えない（パステキスト部分のクリックのみが対象）
- 検索フォルダ一覧の各行に、フォルダごとの詳細設定を開くためのボタンを追加する（詳細は「検索フォルダの詳細設定ダイアログ」を参照）

##### 検索フォルダの詳細設定ダイアログ

- 検索フォルダ一覧の各行に配置したボタンを押下すると、対象フォルダに対する以下3項目を編集できる設定ダイアログ（モーダルまたは展開パネル）を表示する
- 各設定はフォルダ単位で個別に保存する（既存の検索フォルダ登録データに、フォルダごとの詳細設定を追加する形で保持する想定）
- 既存の登録済み検索フォルダに対しても、下記のデフォルト値が自動的に適用される想定とする（既存ユーザーが手動で設定し直す必要はない）

###### 検索階層数

- そのフォルダ配下を何階層まで検索するかを、フォルダごとに数値で指定できる
- デフォルト値：3階層

###### フォルダ自体を検索対象に含めるか

- ON/OFFのトグルで、フォルダ自体（ファイルではなくディレクトリ）を検索結果に含めるかどうかをフォルダごとに指定できる
- デフォルト値：OFF（フォルダは検索結果に表示されず、ファイルのみが検索対象となる）

###### 拡張子フィルタリング

- 「ホワイトリスト」「ブラックリスト」のいずれか一方をフォルダごとに選択する排他方式とする（同時に両方を効かせることはできない）
- ホワイトリスト・ブラックリストいずれも、拡張子をタグ形式（1件ずつ追加・削除できるUI）で登録する
- デフォルトのフィルタモードは「ブラックリスト」、デフォルトのブラックリスト内容は空とする（空の場合は全拡張子を許可として扱う）。つまり初期状態は「制限なし」であり、必要に応じてブラックリストに除外したい拡張子を追加していく運用を想定する
- ホワイトリストモードに切り替えた場合は、タグ追加した拡張子のみを検索対象とする。`*` のような「全許可」を意味する特殊タグは用意しない（ホワイトリストへ切り替えた時点で、明示的にタグを追加しない限り検索対象は0件になる）

#### パス貼り付け

- 機能 ON/OFF トグル（デフォルト ON。OFF 時は `CF_HDROP`・テキスト形式いずれのパス貼り付けでも暗黙判定は発火せず、通常の検索文字列として扱う）
- 機能の詳細（判定方法・表示順序・機能1／機能2の挙動）は「パス貼り付けによる検索フォルダ管理」節を参照

#### 計算・変換

- 数式計算・URLエンコード/デコードの自動表示など、計算・変換系の入力補助機能をまとめるカテゴリ
- カテゴリ内の各機能（数式計算、URLエンコード/デコード等）は、それぞれ独立した「機能ブロック」（見出し＋主設定チェックボックス＋従属設定、の単位）として扱う
- 各機能の主設定チェックボックスの下に、その機能に従属する設定（例: 数式計算の「計算結果をカンマ区切りでコピー」）をインデントして配置する
- 機能ブロック同士は視覚的に区切る（見出しまたは区切り線）
- 主設定チェックボックスが OFF の場合、従属設定はグレーアウトし操作不可にする
- 今後、このカテゴリに新機能（単位変換等）を追加する場合も、同じ機能ブロックの型を踏襲する

##### 機能ブロック: 数式計算

- 機能 ON/OFF トグル（OFF 時は数式入力による計算結果の表示を無効化する。OFF でもファイル検索結果の表示・選択には影響しない）
- 対応する記法：四則演算（`+` `-` `*` `/`）、単項の符号（`+`/`-`）、小数点、括弧 `( )` による優先順位の明示（例: `(1+3)/3`、`2*(3+4)`）
- 数式らしい入力（数字・演算子・括弧・空白・小数点のみで構成され、数字と演算子を1文字以上含む文字列）であっても、ファイル検索は排他にせず従来通り実行する。計算結果はファイル検索結果とは別枠の固定表示領域に表示し、URL エンコード/デコード結果と同様に↑↓キーの選択対象に含める（詳細は「基本動作」節を参照）
- 括弧の対応が取れていない場合（閉じ括弧の不足など）を含め、数式のパースに失敗した場合は、既存のパース失敗時と同様に計算結果表示欄自体を表示しない（ファイル検索結果のみ表示する）
- 「計算結果をカンマ区切りでコピー」トグル（数式計算の主設定チェックボックスに従属する設定としてインデント配置する）
  - ON（デフォルト）：クリップボードにカンマ区切りでコピー（例: 1,000）
  - OFF：カンマなしでコピー（例: 1000）
  - 画面上の計算結果表示は常にカンマ区切り表示のまま変更しない（このトグルはコピー時の書式のみに影響する）

##### 機能ブロック: URLエンコード/デコードの自動表示

- 機能 ON/OFF トグル（OFF 時は URLエンコード/デコードの自動表示を無効化する）
- 「スペースは%20のまま保持する」トグル（主設定チェックボックスに従属する設定としてインデント配置する）
  - OFF（デフォルト）：デコード結果のスペースは半角スペースとして表示される
  - ON：スペースだけは `%20` のまま保持される（スペースをURLの終端と誤認識するアプリ対策）
  - デコード処理のみに影響し、エンコード処理（非ASCII文字入力時のURI変換）には影響しない
  - 対象は半角スペース（`%20`）のみで、`%2B` 等の他の記号は従来通りすべてデコードする

#### システムコマンド

- 機能 ON/OFF トグル（OFF 時は `/shutdown` / `/restart` / `/sleep` のプレフィックスコマンド候補を表示しない）
- シャットダウン・再起動・スリープの3コマンドそれぞれについて、`/` に続くキーワード部分（デフォルト：`shutdown`・`restart`・`sleep`）を個別に変更できる、3つの独立したテキスト入力を設ける（1つの共通プレフィックス設定ではない）。先頭の `/` はここでは変更できない
- 変更しようとしたキーワードが、クリップボード履歴の呼び出しキーワードを含む他の既存キーワードと重複する場合は保存を拒否し、エラーを表示する（「プレフィックスキーワードの重複チェック」節を参照）
- システムコマンド候補が1件以上ある場合、ファイル検索・数式計算・URLエンコード/デコードとは排他になる。この間はファイル検索（`search_files`）を呼び出さず、検索結果リストにはシステムコマンド候補のみを表示する（「システムコマンド機能」節を参照）

#### Web検索

- 機能 ON/OFF トグル（OFF 時は「Googleで〇〇を検索」の固定行を表示しない）

#### クリップボード

- 機能 ON/OFF トグル（デフォルト ON。OFF 時はクリップボードの変化を監視しても履歴に記録せず、呼び出しキーワードにも反応しない）
- 「呼び出しキーワード」欄の変更（デフォルト：`cb`）。先頭の `/` は固定で自動的に付与されるため、変更対象は `/` に続くキーワード部分のみである旨を画面上に明記する
- 変更しようとしたキーワードが、システムコマンドの3キーワードを含む他の既存キーワードと重複する場合は保存を拒否し、エラーを表示する（「プレフィックスキーワードの重複チェック」節を参照）
- 最大保持件数の変更（デフォルト：50件）。入力値の妥当性：1件以上200件以下の正の整数のみを許可し、範囲外・不正な値（小数・文字列・空欄等）が入力された場合は保存せずエラーを表示する（`/recent` の「最大表示件数」と同じ挙動・同じ範囲。詳細は「最近使ったファイル」節を参照）

#### 最近使ったファイル

- 機能 ON/OFF トグル（デフォルト ON。OFF 時は `/recent` のプレフィックスコマンド候補を表示せず、呼び出しキーワードにも反応しない）
- 「呼び出しキーワード」欄の変更（デフォルト：`recent`）。先頭の `/` は固定で自動的に付与されるため、変更対象は `/` に続くキーワード部分のみである旨を画面上に明記する
- 変更しようとしたキーワードが、システムコマンドの3キーワードやクリップボード履歴の呼び出しキーワードを含む他の既存キーワードと重複する場合は保存を拒否し、エラーを表示する（「プレフィックスキーワードの重複チェック」節を参照）
- 「保持期間」欄の変更（デフォルト：180日）。呼び出しキーワードの変更欄と同様、入力欄＋保存ボタンのUIとする。入力値の妥当性：1日以上3650日（10年）以下の正の整数のみを許可し、範囲外・不正な値が入力された場合は保存せずエラーを表示する（詳細は「期間による足切り」節を参照）
- 「最大表示件数」欄の変更（デフォルト：50件）。保持期間と同様、入力欄＋保存ボタンのUIとする。入力値の妥当性：1件以上200件以下の正の整数のみを許可し、範囲外・不正な値が入力された場合は保存せずエラーを表示する（詳細は「表示件数」節を参照）

  2. **GitHubリポジトリ**：ラベル「GitHubリポジトリ」、リンク先 `https://github.com/hidecode365/win-launcher`。クリックでOS標準ブラウザで開く
  3. **X（旧Twitter）**：ラベル「X (@hidecode365)」、リンク先 `https://x.com/hidecode365`。クリックでOS標準ブラウザで開く
  4. **寄付・支援**：ラベル「開発者を応援する（Buy Me a Coffee）」、リンク先 `https://buymeacoffee.com/hidecode365`。クリックでOS標準ブラウザで開く

  - `tauri-plugin-shell` は既にWeb検索機能等で導入済みのため、新規の依存関係追加は不要。ただし将来的に未導入の状態でこの機能を実装する場合は、依存関係として `tauri-plugin-shell` の追加が必要になる

### システム連携

- システムトレイ常駐
- トレイメニューの項目構成は以下の順とする
  - 「Show WinLauncher」：ウィンドウを表示
  - 「Check for Updates」：手動でのアップデートチェックを実行する（起動時チェックと同じロジック。ラベルは英語固定）
  - 「Start with Windows」：自動スタートのON/OFF切替（現在の状態をメニューに表示し、クリックでトグル）
  - 「Restart」：アプリケーションを再起動する（現在のプロセスを終了し、同じ実行ファイルを再度起動する。Tauri の process プラグインの relaunch 機能等を使用する）
  - 「Quit」：アプリケーションを終了する
- トレイアイコンのツールチップは「WinLauncher — {現在のホットキー}」形式で表示する
  - アプリ起動時：plugin-store から保存済みホットキーを読み込み、そのキー文字列でツールチップを設定する（未設定ならデフォルトの Alt+Space）
  - ホットキー変更・保存時：新しいホットキー文字列でトレイのツールチップも即時更新する

### デザイン

- ウィンドウは枠なし・画面中央・角丸
- Windows 11 Fluent ライクなデザイン
- ウィンドウはマウスドラッグで移動可能（位置は永続化せず、非表示→再表示のたびに画面中央に戻る）
- ウィンドウは枠をドラッグしてリサイズ可能。デフォルトサイズ・最小サイズともに幅640px×高さ420pxとする
- ウィンドウサイズは永続化し、アプリ再起動後も最後に設定したサイズを維持する（位置は永続化しない既存仕様は維持する）
- 検索結果のファイルアイコンは Windows のシェルアイコンを取得して表示する（エクスプローラーと同じアイコン）
